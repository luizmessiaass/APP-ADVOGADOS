<objective>
Consent re-gate: backend adds termos_versao_atual to GET /api/v1/tenant/status + TERMS_VERSION config.

Purpose: When the privacy policy changes, all clients must re-accept before accessing process data. The mechanism: backend exposes the current terms version via `GET /api/v1/tenant/status`; app_cliente compares it with the locally stored version at every launch; if different, the consent screen blocks navigation. This plan delivers the backend half (termos_versao_atual in the status response + TERMS_VERSION config) and documents the Android integration point precisely so Plan 04 can wire the Android side.

Output: Extended `GET /api/v1/tenant/status` endpoint with `termos_versao_atual` field + Vitest tests + config.ts TERMS_VERSION env var.

Note: The GET /api/v1/tenant/status endpoint was introduced in Phase 7 (D-11). This plan extends the response. If the Phase 7 route has not yet been created (phases are planned but not all executed), this plan creates it from scratch following the established pattern.

    ```typescript
    import type { FastifyInstance } from 'fastify'
    import { supabaseAsUser } from '../../lib/supabase.js'
    import { env } from '../../config.js'

    export async function tenantRoutes(app: FastifyInstance): Promise<void> {
      /**
       * GET /api/v1/tenant/status
       * Returns tenant subscription status, grace period banner flag,
       * and current terms version for consent re-gate comparison.
       * Phase 7: tenant_status + grace_banner
       * Phase 8 (D-07): + termos_versao_atual
       */
      app.get('/status', async (req, reply) => {
        const token = req.headers.authorization?.slice(7) ?? ''
        const db = supabaseAsUser(token)

        const { data: escritorio, error } = await db
          .from('escritorios')
          .select('status, grace_banner')
          .eq('id', req.user.tenant_id)
          .single()

        if (error || !escritorio) {
          return reply.code(404).send({
            success: false,
            error: 'Escritório não encontrado',
            code: 'NOT_FOUND',
          })
        }

        return reply.send({
          tenant_status: escritorio.status,
          grace_banner: escritorio.grace_banner ?? false,
          termos_versao_atual: env.TERMS_VERSION,  // D-07: consent re-gate version
        })
      })
    }
    ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| app_cliente → GET /api/v1/tenant/status | Authenticated client fetches terms version to compare against local DataStore |
| Server env.TERMS_VERSION → app_cliente | Version string flows from server config to client; tampering would cause false consent bypass |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-8-05 | Tampering | Consent re-gate bypass | mitigate | Comparison uses BOTH local constant AND server `termos_versao_atual`. An attacker would need to control both the deployed app binary AND the backend config to bypass. Server version is the authoritative source (D-07). |
| T-8-06 | Spoofing | Unauthorized access to tenant/status | mitigate | Endpoint is behind the existing auth plugin — req.user.tenant_id is extracted from the verified JWT. No unauthenticated access. |
| T-8-07 | Information Disclosure | termos_versao_atual leaks policy change timing | accept | The version string is an ISO date (e.g., "2026-04-16") — no PII, minimal business sensitivity. Acceptable to expose to authenticated clients. |
</threat_model>
