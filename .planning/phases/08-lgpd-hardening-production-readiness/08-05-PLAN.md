        try {

                _splashState.value = SplashState.Authenticated

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| app_cliente → GET /api/v1/tenant/status | Authenticated client fetches terms version — network failure must not permanently block access |
| DataStore stored version → consent re-gate | Local storage is the only bypass gate — tampering would allow skipping re-consent |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-8-15 | Tampering | Consent re-gate bypass via DataStore manipulation | mitigate | Server version is the authoritative source (D-07). Even if DataStore is cleared, next launch will fetch server version and show consent screen if version has never been accepted for that version string. An attacker controlling DataStore could set TERMS_VERSION_KEY to the expected server version — this is a local-only bypass with no server-side impact. Acceptable risk for v1. |
| T-8-16 | Denial of Service | checkTermsVersion() network failure blocks user permanently | mitigate | Network failures in checkTermsVersion() fall through to Authenticated state (non-blocking). User is not locked out on connectivity failure. The consent screen only appears when version is definitively mismatched — not on network error. |
| T-8-17 | Repudiation | No audit trail for re-consent acceptance | accept | DataStore records the accepted version string. Backend already logs initial consent in lgpd_consentimentos table (Phase 5). Re-acceptance is lighter-weight — stored locally. Acceptable for v1; lawyer review (D-16) may recommend server-side re-consent logging. |
</threat_model>
