<objective>
Implement the BullMQ daily cron job that drives the grace period state machine. Runs at
09:00 BRT (12:00 UTC) daily, processes all tenants in 'grace' or 'read_only' status,
applies state transitions via pure logic from grace-period.ts, and sends transactional
emails via Resend SDK.

Purpose: BILLING-06 requires the automated progression Day 0→Day 3→Day 7→Day 14.
Without this cron, grace periods would never advance — tenants would stay in 'grace'
forever after a payment failure.

Output: grace-period queue/worker files + worker.ts registration + resend@6.12.0 installed.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| BullMQ cron → escritorios table | Cron runs with service role — a bug could mass-update tenant statuses |
| Worker → Resend API | Resend API key is a secret — must be in env vars, never in code |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-7-15 | Tampering | Cron mass-applies wrong transitions | mitigate | Pure function gracePeriodStateTransition is unit-tested independently (Plan 01 stubs); DB writes only happen AFTER function returns valid actions |
| T-7-16 | Tampering | Cron registered in both API and Worker | mitigate | upsertJobScheduler in Worker ONLY. Comments in code explicitly warn against registering in API process |
| T-7-17 | Availability | Resend email fails, status not updated | accept | If email fails after DB update, the status is already correct. Retry happens next day via cron (natural idempotency). Email failure is logged to Sentry but does not block status progression. |
| T-7-18 | Information Disclosure | RESEND_API_KEY in plaintext env | mitigate | Stored in Railway env vars (secret), never committed to git. Listed in user_setup. |
</threat_model>
