- Adicione como GitHub Secret: `FIREBASE_SERVICE_ACCOUNT_JSON`

**4. Verificar .gitignore:**

  <action>
Executar o push da migration para o Supabase remoto:

```bash
# Exportar token de acesso (obtido de .planning/credentials ou memória do projeto)
export SUPABASE_ACCESS_TOKEN="<token-do-projeto>"

# Push da migration
supabase db push --project-ref <project-ref-do-portaljuridico>
```

Se o comando pedir confirmação interativa, confirmar com "y".

**Verificação imediata após push:**
```bash
# Verificar que as tabelas existem
supabase db diff --use-migra --project-ref <project-ref> 2>&1 | grep -E "device_tokens|notifications"
```

Se o push falhar por conflict de migration, verificar se a tabela já existe com:
```sql
-- No Supabase SQL Editor
SELECT table_name FROM information_schema.tables 
WHERE table_schema = 'public' 
AND table_name IN ('device_tokens', 'notifications');
```

**Em caso de erro "already exists":** A migration foi aplicada anteriormente de forma manual. Marcar como aplicada:
```bash
supabase migration repair --status applied 0007_device_tokens_notifications --project-ref <ref>
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| google-services.json → git repo | Arquivo contém project_id e API keys públicas do Firebase; não deve ser commitado |
| service account key → backend env | JSON de alta sensibilidade — comprometimento permite enviar FCM para todos os devices |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-10 | Information Disclosure | google-services.json commitado no git | mitigate | Adicionar ao .gitignore antes do primeiro commit; verificar com `git status app-cliente/google-services.json` |
| T-06-11 | Information Disclosure | serviceAccountKey.json commitado no git | mitigate | Jamais commitar o arquivo; usar apenas via env var no .env (que também está no .gitignore) |
| T-06-12 | Denial of Service | Schema push aplicado no banco errado (staging vs prod) | mitigate | Verificar `project-ref` no supabase/config.toml antes do push; output do comando confirma o projeto |
</threat_model>
