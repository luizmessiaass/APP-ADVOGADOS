<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Android client → POST /api/devices/register | Token FCM vem do cliente; deve ser vinculado apenas ao user_id autenticado |
| Backend FCM service → Firebase API | Service account JSON é segredo de alto valor |
| GET /api/notifications → Supabase | Query deve ser escopo ao user_id do JWT, nunca cross-tenant |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-04 | Information Disclosure | FCM token em logs | mitigate | Nunca incluir `token` no payload de log; usar apenas `deviceId` para rastreabilidade |
| T-06-05 | Elevation of Privilege | POST /api/devices/register sem tenant check | mitigate | `tenant_id` extraído do JWT `app_metadata` pelo middleware, nunca do body do request |
| T-06-06 | Information Disclosure | FIREBASE_SERVICE_ACCOUNT_JSON em código ou git | mitigate | Inicializar via `process.env.FIREBASE_SERVICE_ACCOUNT_JSON`; `.gitignore` deve incluir `serviceAccountKey.json`; documentar no user_setup |
| T-06-07 | Tampering | deep_link injection via dados de movimentação | mitigate | `processoId` é UUID gerado pelo backend (Supabase gen_random_uuid()); nunca concatenar texto livre no deep_link |
| T-06-08 | Information Disclosure | GET /api/notifications cross-tenant | mitigate | Query sempre com `.eq('user_id', userId)` extraído do JWT; RLS da tabela como segunda linha de defesa |
| T-06-09 | Denial of Service | FCM dispatch bloqueando worker principal | accept | firebase-admin.send() é async/await; em caso de erro non-token, relança para BullMQ retry com backoff |
</threat_model>
