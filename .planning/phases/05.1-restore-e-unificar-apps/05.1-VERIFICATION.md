### Human Verification Required

#### 1. Login as advogado — EscritorioNavGraph mounted

**Test:** On emulator or device, install the debug APK. Log in with valid advogado credentials.
**Expected:** After successful login, the app displays the ClienteListScreen (EscritorioNavGraph is active), not ProcessoListScreen.
**Why human:** Role-based NavGraph selection is a runtime branch involving Hilt injection, Compose recomposition, and StateFlow collection. Code analysis confirms the wiring is correct but a real run is required to confirm no crash at EscritorioNavGraph's `authRepository` parameter injection.

#### 2. App restart with saved cliente token

**Test:** Log in as cliente, go to ProcessoListScreen, force-stop the app, reopen.
**Expected:** App reopens directly to ProcessoListScreen (bypassing login), confirming DataStore persistence of TOKEN_KEY and ROLE_KEY.
**Why human:** DataStore persistence across process death requires a live device test. The SplashViewModel flow logic is correct in code.

#### 3. App restart with saved advogado token

**Test:** Log in as advogado, force-stop the app, reopen.
**Expected:** App reopens directly to EscritorioNavGraph (ClienteListScreen), bypassing login.
**Why human:** This tests the two-StateFlow read in MainActivity (`startDestination` + `userRole`). There is a potential race condition: both flows have `initialValue = null` and emit independently. If `role` emits null before the stored value arrives, the `else` branch (ClienteNavGraph) could briefly flash. Needs live testing to confirm the guard `if (startDest != null)` is sufficient.

#### 4. Pre-existing build warning: core-ui/Type.kt Google Fonts R.array

**Test:** Run `./gradlew :app-cliente:compileDebugKotlin` and check the exit code.
**Expected:** Exit 0. The SUMMARY notes this as a pre-existing issue; confirm whether it was fixed or is still blocking.
**Why human:** Gradle build cannot be run in verification without the full Android SDK environment.
